# PALL-FIN: Project Overview & Architecture

**PALL-FIN** is a high-performance, parallel MapReduce-style financial data aggregation system. It was migrated from a sequential Python implementation to a highly optimized C backend to achieve significant speedups when processing millions of financial transaction records.

The system processes large CSV datasets (e.g., 10M+ rows of stock trades), calculates aggregate buy/sell volumes per ticker, and determines net market flows. It uses POSIX threads (`pthreads`) for multi-core parallelization and a **Binary Tree Reduction** algorithm for merging worker results in $O(\log W)$ time.

---

## 1. System Architecture

The core architecture follows a classic **Divide, Map, and Reduce** paradigm, heavily optimized for I/O and memory efficiency:

```mermaid
graph TD
    A[data/stocks.csv] -->|O(1) memory mapping| B(Divide Phase)
    
    subgraph Parallel Map Phase
    B -->|Byte offsets| C1[Worker Thread 0]
    B -->|Byte offsets| C2[Worker Thread 1]
    B -->|Byte offsets| C3[Worker ... Thread N]
    end
    
    subgraph Binary Tree Reduce Phase
    C1 --> D1[Merge Level 1]
    C2 --> D1
    C3 --> D2[Merge Level 1]
    D1 --> E[Global Final Merge]
    D2 --> E
    end
    
    E -->|JSON Output| F(results.json)
    F --> G[Interactive UI Dashboard]
```

### Key Algorithmic Features
1. **Zero-Copy / Byte-Range Partitioning (Divide):** Instead of loading the entire 200MB+ CSV into RAM, the main thread reads the file size, calculates byte-range chunks for each worker, and adjust pointers to the nearest newline boundary. This avoids out-of-memory errors and keeps the divide phase $O(1)$.
2. **Parallel Map (Workers):** Each pthread parses its specific chunk of the file independently and builds a local hashmap-like structure (`StockSummary` array) of ticker volumes.
3. **Binary Tree Reduction (Reduce):** Instead of one thread merging all $N$ worker results sequentially (which takes $O(W)$ time and causes bottlenecks), workers merge their results in a tournament-tree fashion. Depth of the tree is $\log_2(W)$.

## 2. Codebase Breakdown: What Code Does What

### Backend (C Engine)
* **`benchmark.c`**
  * **Role:** The main entry point. 
  * **Function:** It manages the execution flow. It runs both the sequential baseline and the parallel engine, records the high-resolution execution times (via `clock_gettime`), verifies that both outputs match identically (Accuracy Check), and finally exports the output to `dashboard/results.json`.
* **`src/aggregator_core.c` & `src/aggregator_core.h`**
  * **Role:** Core data structures and sequential logic.
  * **Function:** Defines the `StockSummary` struct (holding total buys, total sells, net flow). Contains `summary_merge()` which safely combines two summaries together. It also houses the `sequential_aggregate()` function, acting as the baseline to measure parallel speedup against.
* **`src/parallel_engine.c` & `src/parallel_engine.h`**
  * **Role:** The multi-threading powerhouse.
  * **Function:** Spawns POSIX threads. Contains the `worker_map()` function where individual threads parse CSV chunks using fast pointer arithmetic (`strtok_r`). Implements the synchronization barriers (`pthread_barrier_t`) and the loop logic for the Binary Tree Reduction algorithm.

### Analysis & Tooling
* **`Makefile`**
  * **Role:** Build system.
  * **Function:** Compiles the C binaries with `-O2` speed optimizations and links the `pthread` library. Handles `make run` to execute the benchmark and `make clean`.
* **`src/data_generator.py`**
  * **Role:** Synthetic dataset creation.
  * **Function:** A Python script that uses `yfinance` to seed realistic stock tickers and generates massive mock CSV transaction datasets for testing.
* **`scaling_analysis.py`**
  * **Role:** Performance profiler.
  * **Function:** Automates running the C benchmark against increasing dataset sizes (100K to 10M rows). Calculates the **Parallel Crossover Point** and **Amdahl's Parallel Fraction**, exporting these insights to `dashboard/scaling_results.json`.

### Frontend Dashboard
* **`dashboard/index.html`** & **`dashboard/style.css`**
  * **Role:** User Interface.
  * **Function:** A dark-mode, glassmorphism-styled terminal UI that visually represents the MapReduce algorithm and system metrics.
* **`dashboard/main.js`**
  * **Role:** UI Logic.
  * **Function:** Fetches `results.json` and `scaling_results.json` generated by the backend. It dynamically populates the HTML with the runtime stats, algorithm visualizers, and renders a Chart.js doughnut chart of the market distribution.

### Legacy (Archive)
* **`archive/`**
  * **Role:** Code Graveyard.
  * **Function:** Contains the original Python-based parallel implementation (`aggregator_core.py`, `parallel_engine.py`, `benchmark.py`). This was kept for historical reference and to measure how much faster the C implementation is (~2x faster base, scaling much better).

---

## 3. Data Flow

1. **Generation:** `make gen` runs `data_generator.py` to create `data/stocks.csv`.
2. **Execution:** `make run` executes the compiled `./financial_agg`.
3. **Processing:** The C code splits the file, maps data on 8 cores, reduces it, and outputs JSON.
4. **Analysis:** `python3 scaling_analysis.py` can be run to test various dataset sizes.
5. **Visualization:** `cd dashboard && npm run dev` loads the JSON files and presents the analytics to the user.
